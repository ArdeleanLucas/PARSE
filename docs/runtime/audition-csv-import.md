# Adobe Audition CSV speaker import

`POST /api/onboard/speaker` treats an uploaded `csv` file as an Adobe Audition marker export when concepts-style parsing (`id,concept_en`) returns no rows and the header contains `Name` and `Start` columns after case-insensitive trimming. Audition imports accept tab, comma, or semicolon delimiters, UTF-8 with or without BOM, and cue times in `H:MM:SS.mmm`, `M:SS.mmm`, or plain-seconds form. The `Name` field may use a parenthesized hierarchical prefix such as `(8.4)- to listen to`, a square-bracket hierarchical prefix such as `[5.1]- The boy cut the rope`, or a plain integer prefix such as `9- nine`; when a prefix is parsed, terminal `A`/`B`/`C`/`D` variant markers are stripped from the label while the parsed prefix is retained as Audition metadata, not as the PARSE concept id. On onboarding, each cue label is resolved against existing `concepts.csv` rows by case-insensitive `concept_en` match first; only unmatched labels allocate the next available integer id, so the registry remains integer-id only. Valid Audition rows are appended in CSV row order to `tiers.concept.intervals` and mirrored to `tiers.ortho_words.intervals` with the CSV timestamps preserved exactly apart from the duration-derived end time fallback of `start + 1.0` when duration is missing or non-positive. Each imported interval stores additive trace metadata: `concept_id` is the resolved integer PARSE id, `import_index` is the zero-based physical CSV row position after the header, and `audition_prefix` is the original parsed prefix such as `8.4`, `9`, or `32`, or a synthetic `row_<import_index>` value for rows with no parseable prefix. Duplicate prefixes or duplicate resolved concept ids intentionally produce multiple intervals, each with its own `import_index`, because variant pronunciations and repeated elicitations are legitimate linguistic observations. This import intentionally does not populate or mutate `tiers.ortho`, `tiers.ipa`, or `tiers.bnd`; those remain downstream STT, IPA, and boundary-refinement outputs.

Rows with no parseable prefix are still imported: PARSE uses the verbatim `Name` as the interval label and assigns `audition_prefix="row_<import_index>"` so the source row remains traceable without pretending the synthetic value is a structural id. Malformed prefix attempts, such as `(2.29- child of one's son)-`, follow the same bare-row path. Downstream consumers must treat every `audition_prefix` as an opaque source marker rather than parsing it for hierarchy or numeric meaning.

`concepts.csv` now writes the current schema `id,concept_en,source_item,source_survey,custom_order`. Audition onboarding stores the parsed cue prefix in `source_item` for resolved concept rows (`(1.2)- forehead` -> `source_item=1.2`), while `source_survey` remains an unconstrained attribution string for survey/source families such as `KLQ`, `EXT`, or `JBIL`. Readers tolerate older two-column `id,concept_en` files and the legacy `survey_item` header, but every backend writer emits the current five-column header on the next write.

The same endpoint also accepts an optional second multipart field named `commentsCsv` alongside `csv`. A comments CSV must be the Audition companion file for the cue CSV: identical parsed row count and physical row order, with analyst notes appended only to the `Name` column. The backend stores that upload beside the cue file as `<cue-stem>.comments<cue-suffix>`, parses both files, pairs rows by zero-based row index, strips the cue row's exact `raw_name` prefix from the comments row's `raw_name`, trims leading whitespace plus `-`, `:`, `–`, or `—` connectors, and writes the remaining text to `parse-enrichments.json` under `lexeme_notes[<speaker>][<resolved integer concept_id>].import_note` with `import_raw`, `import_index`, `audition_prefix`, and `updated_at` trace fields. `commentsCsv` without `csv` is rejected with HTTP 400. If the parsed row counts differ, PARSE logs a `[audition-csv] comments row count mismatch ...` message to stderr and skips only the notes import; if one row's comments `Name` does not start with the cue `Name`, PARSE logs `[audition-csv] row <i> misaligned ...`, skips that row, and continues importing later notes. Cue-only uploads still import lexeme intervals and concepts normally without deriving notes from the cue labels themselves.

API round-trips preserve additive trace metadata: `GET /api/annotations/<speaker>` and `POST /api/annotations/<speaker>` return or accept intervals carrying optional `concept_id`, `import_index`, `audition_prefix`, `conceptId`, and `source` fields. Server-side normalization explicitly enumerates these passthrough fields; arbitrary unknown interval fields are still dropped. The on-disk interval arrays may be sorted by start time on save, but CSV row order can always be reconstructed from `import_index`.

Every concept-tier interval saved through `POST /api/annotations/<speaker>` is now normalized to carry an integer `concept_id` when it has non-empty text. The save-time gate resolves labels by case-insensitive, whitespace-normalized `concept_en` match against `concepts.csv`; unknown labels allocate the next integer id and append it to the registry. The frontend matcher uses `concept_id` equality only; interval text is display metadata and is never compared. There is no bulk migration: legacy concept intervals that were never re-imported or saved remain without `concept_id` and render as unannotated until a CSV-only reimport or an Annotate UI save causes the gate to fire.

## CSV-only MCP reimport and revert

For already-onboarded speakers, PARSE exposes two paired MCP/chat tools: `csv_only_reimport` and `revert_csv_reimport`. `csv_only_reimport` accepts `speaker`, `sourceCsv`, optional `commentsCsv`, and optional `dryRun`; it never accepts a WAV path and never re-uploads audio. Instead, it resolves the speaker's existing primary WAV from `source_index.json` (preferring `is_primary=true`, otherwise the first registered source) and then reuses the server-side Audition CSV worker against that registered audio. Every non-dry-run reimport first captures a mandatory backup at `<project_root>/annotations/backups/<YYYYMMDDTHHMMSSZ>-<speaker>-csv-reimport/` containing any present `<speaker>.json`, `<speaker>.parse.json`, `parse-enrichments.json`, `concepts.csv`, plus `manifest.json` describing captured files, original CSV inputs, resolved WAV path, and the worker result. `revert_csv_reimport` restores only the files listed in that backup manifest, either from a named `backupDir` under `annotations/backups/` or from the latest matching `*-<speaker>-csv-reimport` directory when omitted; concept ids or other project edits made outside those captured files are not globally undone. Backup pruning/retention is intentionally not implemented yet; backups remain indefinitely until a future policy is chosen.
