# Adobe Audition CSV speaker import

`POST /api/onboard/speaker` treats an uploaded CSV as an Adobe Audition marker export when concepts-style parsing (`id,concept_en`) returns no rows and the header contains `Name` and `Start` columns after case-insensitive trimming. Audition imports accept tab, comma, or semicolon delimiters, UTF-8 with or without BOM, and cue times in `H:MM:SS.mmm`, `M:SS.mmm`, or plain-seconds form. The `Name` field may use either a parenthesized hierarchical prefix such as `(8.4)- to listen to` or a plain integer prefix such as `9- nine`; terminal `A`/`B`/`C`/`D` variant markers are stripped from the label while the parsed prefix is retained as Audition metadata, not as the PARSE concept id. On onboarding, each cue label is resolved against existing `concepts.csv` rows by case-insensitive `concept_en` match first; only unmatched labels allocate the next available integer id, so the registry remains integer-id only. Valid Audition rows are appended in CSV row order to `tiers.concept.intervals` and mirrored to `tiers.ortho_words.intervals` with the CSV timestamps preserved exactly apart from the duration-derived end time fallback of `start + 1.0` when duration is missing or non-positive. Each imported interval stores additive trace metadata: `concept_id` is the resolved integer PARSE id, `import_index` is the zero-based physical CSV row position after the header, and `audition_prefix` is the original parsed prefix such as `8.4`, `9`, or `32`. Duplicate prefixes or duplicate resolved concept ids intentionally produce multiple intervals, each with its own `import_index`, because variant pronunciations and repeated elicitations are legitimate linguistic observations. This import intentionally does not populate or mutate `tiers.ortho`, `tiers.ipa`, or `tiers.bnd`; those remain downstream STT, IPA, and boundary-refinement outputs. Companion comments-file row-index joining is a follow-up and is not part of this contract.
