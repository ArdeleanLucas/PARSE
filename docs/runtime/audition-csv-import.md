# Adobe Audition CSV speaker import

`POST /api/onboard/speaker` treats an uploaded `csv` file as an Adobe Audition marker export when concepts-style parsing (`id,concept_en`) returns no rows and the header contains `Name` and `Start` columns after case-insensitive trimming. Audition imports accept tab, comma, or semicolon delimiters, UTF-8 with or without BOM, and cue times in `H:MM:SS.mmm`, `M:SS.mmm`, or plain-seconds form. The `Name` field may use either a parenthesized hierarchical prefix such as `(8.4)- to listen to` or a plain integer prefix such as `9- nine`; terminal `A`/`B`/`C`/`D` variant markers are stripped from the label while the parsed prefix is retained as Audition metadata, not as the PARSE concept id. On onboarding, each cue label is resolved against existing `concepts.csv` rows by case-insensitive `concept_en` match first; only unmatched labels allocate the next available integer id, so the registry remains integer-id only. Valid Audition rows are appended in CSV row order to `tiers.concept.intervals` and mirrored to `tiers.ortho_words.intervals` with the CSV timestamps preserved exactly apart from the duration-derived end time fallback of `start + 1.0` when duration is missing or non-positive. Each imported interval stores additive trace metadata: `concept_id` is the resolved integer PARSE id, `import_index` is the zero-based physical CSV row position after the header, and `audition_prefix` is the original parsed prefix such as `8.4`, `9`, or `32`. Duplicate prefixes or duplicate resolved concept ids intentionally produce multiple intervals, each with its own `import_index`, because variant pronunciations and repeated elicitations are legitimate linguistic observations. This import intentionally does not populate or mutate `tiers.ortho`, `tiers.ipa`, or `tiers.bnd`; those remain downstream STT, IPA, and boundary-refinement outputs.

The same endpoint also accepts an optional second multipart field named `commentsCsv` alongside `csv`. A comments CSV must be the Audition companion file for the cue CSV: identical parsed row count and physical row order, with analyst notes appended only to the `Name` column. The backend stores that upload beside the cue file as `<cue-stem>.comments<cue-suffix>`, parses both files, pairs rows by zero-based row index, strips the cue row's exact `raw_name` prefix from the comments row's `raw_name`, trims leading whitespace plus `-`, `:`, `–`, or `—` connectors, and writes the remaining text to `parse-enrichments.json` under `lexeme_notes[<speaker>][<resolved integer concept_id>].import_note` with `import_raw`, `import_index`, `audition_prefix`, and `updated_at` trace fields. `commentsCsv` without `csv` is rejected with HTTP 400. If the parsed row counts differ, PARSE logs a `[audition-csv] comments row count mismatch ...` message to stderr and skips only the notes import; if one row's comments `Name` does not start with the cue `Name`, PARSE logs `[audition-csv] row <i> misaligned ...`, skips that row, and continues importing later notes. Cue-only uploads still import lexeme intervals and concepts normally without deriving notes from the cue labels themselves.
